# Dash Voice Bot — Authentication Tools

Backend tools for the Enterprise Bot Dash voice bot: account balance lookup with OTP-based authentication.

## Tech stack
- Java 17, Spring Boot 3.5
- H2 in-memory database (seeded with 3 test accounts on startup)
- Caffeine cache for OTP (1-minute TTL) and session state (2-minute sliding inactivity timeout)
- SMS sending is mocked (logged to console) — swap in a real provider (e.g. Twilio) later

## Tools

### 1. Verify_Account — POST /api/tools/verify-account
Input: `{ "accountNumber": "1001234567" }`
- Looks up the account. If found, generates a 4-digit OTP, caches it against the account number for 1 minute, and (mock) sends it via SMS.
- Response: `{ "status": "OTP_SENT", "message": "..." }` or `{ "status": "ACCOUNT_NOT_FOUND", "message": "..." }`

### 2. Verify_Otp — POST /api/tools/verify-otp
Input: `{ "accountNumber": "1001234567", "otp": "1234", "sessionId": "<conversation id>" }`
- Validates the OTP against the cache. On success, marks the session authorized (`sessionCache`) and returns the account balance. OTP is single-use (invalidated after success).
- Response: `{ "status": "SUCCESS", "message": "...balance..." }` or `{ "status": "OTP_MISMATCH", "message": "..." }`

### 3. (Optional) GET /api/tools/session-status?sessionId=
Checks whether a session is already authenticated, so the bot doesn't re-trigger authentication mid-session.

## Design choices
- **H2 + Caffeine** instead of an external DB/Redis: fastest to set up with zero external dependencies, sufficient for a demo of this scope.
- **PIN-free flow**: chose Account Number + OTP (not PIN) as the authentication method, since OTP is single-use and time-bound, which is a stronger security pattern than a static PIN for a voice channel.
- **Session ID**: expected to be supplied by the AIDA platform's own conversation/session identifier on each tool call, not generated by this backend.
- **Agentic design**: the LLM (via AIDA) decides when to call these tools based on the Response Guidelines document — no hardcoded conversation flow in the backend.
